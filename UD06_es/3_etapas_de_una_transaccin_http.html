3. Etapas de una transacción HTTP

Con la intención de conocer con más profundidad el protocolo HTTP podemos evaluar en qué consiste una transacción HTTP.

Cada vez que un cliente hace una petición a un servidor, se ejecutan una serie de acciones:

  1. Un usuario accede a una dirección de Internet (URL) seleccionando un enlace de un documento HTML o introduciéndola directamente en la barra de navegación de un navegador web desde la perspectiva del cliente web. El cliente web descodifica la dirección de Internet (URL) separando las diferentes partes. Es así como se identifican el protocolo de acceso , el nodo expresado con el nombre de dominio o su dirección IP, el posible puerto opcional (el valor predeterminado es el 80) y el objeto del servidor requerido.
  2. Se abre una conexión TCP / IP con el servidor llamando el puerto TCP correspondiente. Se hace la petición. En consecuencia, se envían la orden necesaria (GET, POST, HEAD, etc.), la dirección del objeto requerido (el contenido de la dirección de Internet del servidor), la versión del protocolo HTTP utilizada ( en la mayor parte de las ocasiones es HTTP / 1.0) y un conjunto variable de información que incluye datos sobre las capacidades del navegador web, datos opcionales para el servidor, etc.
  3. El servidor localiza el recurso solicitado y devuelve la respuesta al cliente.
  4. Esta respuesta consiste en un código de estado y el tipo de dato con extensiones multipropósito de correo de Internet (MIME, Multipurpose Internet Mail Extension ) de la información de vuelta, seguido de la misma información.
  5. El cliente formatea y muestra el recurso recibido.
  6. Se cierra la conexión TCP.

Este proceso se repite en cada acceso que se haga el servidor HTTP. Por ejemplo, si se recoge un documento HTML que contiene cuatro imágenes, el proceso de transición mostrado con anterioridad se repite cinco veces, es decir, una por el documento HTML y cuatro para las imágenes.

 
Etapas en la transición HTTP



extensión MIME

El protocolo HTML fue diseñado para transportar por la red archivos en formato ASCII. Con el progreso de las tecnologías y la inclusión de diferentes tipos de archivos en un formato diferente al ASCII (imágenes, vídeos, sonidos, etc.) en las aplicaciones por Internet, surgió la necesidad de transformar estos formato ASCII para su correcta recepción en el navegador web.

En 1992, se crearon los tipos MIME, que eran especificaciones para dar formato a los mensajes en un formato diferente al ASCII para que pudieran ser enviados e interpretados por Internet. Más adelante se aplicaron también a los documentos web.

Nomenclatura de tipo

MIME asigna un nombre a cada tipo de datos. Los nombres siguen el siguiente formato:

tipo / subtipo (tipo como subtipo son cadenas de caracteres)

El tipusdefine la categoría general de los datos y el subtipusdefine un tipo más específico de estos datos. El tipo tener los siguientes valores:

  • texto : Indica que el contenido es texto plano. Ejemplos de subtipos: html, xml
  • multiparte : Indica que tiene múltiples partes de datos independientes. Ejemplos de subtipos: form-data, digest
  • message : Para encapsular un mensaje existente. Por ejemplo cuando queremos responder a un mensaje de correo incorporando el mensaje origen. Ejemplos de subtipos: partial, RFC822
  • image : Indica que es una imagen. Ex de subtipos: png, gif
  • audio : Indica que es un audio. Ejemplos de subtipos: mpeg, avi
  • video : Indica que es un vídeo. Ejemplos de subtipos: mp4, 32kadpcm
  • application : Indica que se trata de datos de aplicación de los cuales pueden ser binarios. Ejemplos de subtipos: json, pdf
